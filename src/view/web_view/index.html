How to read the result
The analyzer fetches /robots.txt from the URL and runs three checks:

1. Tier (RQ1) — assigns a defense tier 1–5 based on which bot layers are blocked and how.

2. Conflicts (RQ2) — detects five classes of directive contradictions that make protection semantically ineffective despite its appearance.

3. Compliance score (RQ3) — weighted 0.0–1.0 across three bot layers: APP layer 35%, INFRA layer 45%, Google AI 20%. A site blocking only GPTBot scores 0.35, not 1.0.

4. Improvement advisor — after the result, a targeted guide explains exactly what to change and shows ready-to-copy robots.txt snippets for your specific situation.
Thesis context
About this Tool

What this thesis does

This tool is the artifact of the bachelor thesis "Semantic Configuration Analysis of robots.txt for EU AI Act Compliance" (2DV50E, Linnaeus University, VT 2026). It investigates whether European news publishers are effectively protecting their content from AI training data crawlers using the Robots Exclusion Protocol.

Most existing tools check robots.txt in a binary way — a bot is either blocked or it is not. This tool performs semantic analysis: it checks whether the configuration is internally consistent under RFC 9309 parsing rules and whether the protection actually holds, not just on the surface.

What the results mean

Compliance status
COMPLIANT All three bot layers are blocked and no conflicts cancel that protection.
The robots.txt file is semantically effective — not just syntactically present. All crawler categories (AI assistants, infrastructure harvesters, and Google AI) are denied access and no internal Allow directives override those blocks. This is the only status that represents genuine protection.
NOMINAL The Enumeration Fallacy — protection that appears real but is not.
The site has written blocking rules but they are cancelled by competing Allow directives. Under RFC 9309, when an Allow and Disallow cover the same path at equal specificity, Allow always wins — silently. A correctly behaving crawler reads this file as granting full access, even though Disallow entries are present. Binary tools would classify this site as protected. The SCA does not. The fix is almost always a single directive removal.
PARTIAL Some crawler categories are blocked but not all.
The compliance score reflects which layers are covered. Blocking GPTBot (APP layer, 35%) while leaving CCBot unblocked still means the infrastructure harvesters responsible for most large-scale training data collection have full access. Partial coverage is better than nothing but leaves real gaps. The score shows exactly how large those gaps are.
NON_COMPLIANT No AI protection exists.
The robots.txt file either has no AI-related directives or is absent. Every crawler — from the user-facing AI assistants to the background training harvesters scraping the web at scale — has unrestricted access. The content on this site is freely available for use in AI model training. A single line of robots.txt is sufficient to change this.
Compliance score (0.0 to 1.0)

A weighted score across three crawler layers. The weights reflect the layers' actual contribution to AI training data collection. The INFRA layer (CCBot, Bytespider etc.) carries the highest weight at 45% because these crawlers are responsible for the majority of bulk training data harvesting — they are less visible than GPTBot but significantly more consequential in volume. A site that only blocks the APP layer scores 0.35, not 1.0, because the dominant data collection pathway is still open.

Conflict types — why more rules can mean less protection
WILDCARD_OVERRIDE
HIGH		 A wildcard Allow: / appears in the same section as specific Disallow entries. Because the wildcard matches every path, it always ties with or beats the specificity of any Disallow, causing the Allow to win. This is the classic Enumeration Fallacy pattern — the site looks protected but every named bot has full access.
DUPLICATE_SECTION
MEDIUM		 The same User-agent appears in more than one section. RFC 9309 requires parsers to merge all matching sections but some crawlers only process the first or last match. If one section blocks and another allows, the outcome depends on which section the crawler reads. Consolidate into one block per agent.
ALLOW_DISALLOW
HIGH		 An Allow and Disallow directive cover the same path for the same agent. RFC 9309 specifies that when specificity is equal, Allow wins. If both point to /, Allow wins silently — even though the intent is to block. Removing the Allow entry fixes this immediately.
PARTIAL_BLOCK
LOW		 A Disallow: / appears alongside an Allow: / of identical path length in the same section. Because they tie on specificity, Allow wins by protocol default. The bot receives access despite the presence of a Disallow entry.
EMPTY_DISALLOW
HIGH		 An empty Disallow: directive (no path) means "allow everything" in RFC 9309. When this appears alongside real blocking rules, it functions as a silent allow-all that a compliant crawler must honour, nullifying all other Disallow entries in that section.

How this differs from other tools

Existing binary parsers

Standard tools check whether a specific bot appears with a Disallow directive. They answer "Is GPTBot blocked?" with yes or no. They do not check whether the block is semantically effective, whether other directives cancel it, or whether the INFRA layer is also covered.

This tool — semantic analysis

The SCA parses robots.txt into a structured section map and evaluates it as a semantic unit. It checks all five conflict classes, evaluates three distinct bot layers with weighted importance, and distinguishes between intended and effective protection. A site can appear protected while being fully open.

The Improvement Advisor

After each analysis the tool generates a personalised improvement guide: it identifies exactly what is wrong, explains why it matters, recommends the upgrade path, and provides ready-to-copy robots.txt snippets tailored to the site's current tier. No other parser produces actionable repair instructions.

Tier 3 Surgical No wildcard. Explicit named blocks for both APP-layer and INFRA-layer bots. Effective when maintained, but requires regular updates as new crawlers emerge. COMPLIANT
Tier 2 Porous Only APP-layer bots blocked. INFRA-layer crawlers still have full access. Blocks the visible, named AI assistants while missing the infrastructure harvesters doing most of the data collection. PARTIAL
Tier 1 Open No AI-specific blocking. All crawlers have unrestricted access. No opt-out signal exists in the robots.txt file. NON-COMPLIANT
